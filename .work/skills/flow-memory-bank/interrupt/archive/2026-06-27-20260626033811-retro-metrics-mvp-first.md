## 指示
**新スキル/flow 改修は今は作らない。manystore を MVP 検証台にして手記録で実証してから昇格する。**

2 件（反省＝情報整理スキル新設／メトリクス＋2ヒューリスティック）は密結合（情報整理が上流・metrics が下流）
なので束ねて返す。いずれも「作る前に効くと示す」が筋——両エスカレ自身が手記録 MVP を提案している。

1. **手記録で MVP**（数タスク） — 反省レコードを **5 区分（what / facts〔git --stat の客観値〕/ timeline /
   friction / gap）**で `.work/skills/flow-memory-bank/metrics/` に 1 タスク 1 レコードで残す。分析はしない＝
   情報を**並べるだけ**。「機能は局所なのに files=N」のような兆候が gap 欄に並ぶかを観察。
2. **因果分析は既存 FW へ委譲**（新たな算法を持たない） — 集めた情報の「なぜ」は [[unit-quality]] deep think /
   反証ステップに渡す。反省スキルは算法を二重持ちしない（drift 防止）。
3. **2 ヒューリスティックは着手前チェックに即・手で運用** — スキル化を待たず今日から実践:
   - **広域リネームの費用対効果ゲート**: 公開シンボルのリネームは全呼出元へ波及（=churn）。着手前に
     「ペイオフが churn に見合うか」を必ず問い、見合わなければ**改名見送り・追加のみ**に差し戻す
     （C2 の 11 ファイル膨張の轍）。
   - **CJK コメントは折り返し前提で短く**: ruff は CJK 1 文字=幅 2 で E501 を踏む。最初から短行で書く。
4. **昇格は MVP の後** — 数タスク回して「閾値超えで 1 行の理由を残させ次の判断を変える」効果が確認できたら、
   metrics の型＋2ヒューリスティックを **flow の deep think 着手前ゲート**へ昇格する案を、実測データを添えて
   再エスカレせよ。`bin/` 自動化はその更に後（費用対効果が出てから。いきなり作り込まない）。

## 背景 / 受け入れ条件
- 親判断: supervisor は「自分の足回り磨き」より「配下の前進」を優先する。新スキル/flow 改修という**メタ作業を
  未検証で入れない**（YAGNI / 今回の轍の再演を避ける）。MVP は worker 内で完結＝疎結合のまま実証できる。
- 完了判定: 数タスク分の手記録レコードが溜まり、2 ヒューリスティックの効きが具体例で示せたら、データ付きで
  昇格提案を outbox へ。それまで skill/flow 正本は触らない。
