### 1. 改善計画：工程に「計測→挙動変更」を盛り込む

タスクごとに以下を Memory Bank（`metrics/log.jsonl`・1 タスク 1 行・追記）へ記録したい。
**数字を貯めるのが目的ではなく、閾値超えで 1 行の理由を残させ次の判断を変える**のが本質
（貯めるだけは YAGNI 違反）。

- **汚染される指標は捨てる**：壁時計時間は承認待ち・人間の思考・離席が混入＝effort 指標にならない。
- **クリーンに取れる**：変更ファイル数・行数（git --stat で厳密）／ツール呼出数・最初の編集までの
  Read/Grep 数（=「情報をたどる難しさ」の代理）／品質ゲート再試行回数（=手戻り）／質問回数（transcript）。
- **承認ギャップは除去できる**：transcript のタイムスタンプで、ステップ間隔が閾値（例 60s）超の区間を
  人間 idle と判定し差し引く→「idle 除去後アクティブ時間」が妥当な近似。**承認があっても正しい時間は取れる。**
- **段階**：MVP＝手記録（git stat は正確・活動系はざっくり・flags 必須）を数タスク試す → 効くと分かってから
  `bin/memory-bank-metrics`（git 範囲＋transcript パース）で自動化。**いきなり自動化を作り込むのは反対**
  （費用対効果未確認のまま作り込む＝今回の轍の再演）。
- **昇格先**：これは flow スキル本体（project 非依存）の話。各 repo で MVP 検証後、skill へ昇格＋bin 自動化。

### 3. 何に時間がかかったか（言い訳＝率直な根本原因）

- **C2 が 11 ファイルに膨張**：`create_key_value_store` は connect/service/config/README/tests と
  6 箇所から使う**公開シンボル**。ディレクトリ構造は*機能変更*を局所化できるが、*公開 API のリネーム*は
  定義上すべての呼び出し元へ波及する＝**構造の失敗ではなく、リネームの性質**。C1（局所＝code4）は構造が効いた。
- **判断ミス（ここが本丸）**：ユーザーが「生口はトップ公開に残す」を選んだ時点で C2 の主目的（safe を既定に
  する格下げ）は消え、残ったのは「名前に unsafe を付ける」だけ。それを**11 ファイルの破壊的リネームで
  買った**のは割に合わない。決定の場で「格下げしないなら改名は見送り、`create_safe_*` の追加だけ」と
  **もっと強く差し戻すべきだった**。→ 横断ヒューリスティック化したい：**「広域リネームは、それ自体の
  ペイオフが churn に見合うかを着手前ゲートで必ず問う」**。
- **CJK の E501 reflow ループ**：新ファクトリ全部に多行 JP docstring を書き、CJK は ruff で 1 文字=幅 2 の
  ため E501 に何度も引っかかり reflow を繰り返した＝時間の主因。**最初から短い行で書けば避けられた無駄。**
  → flow/quality へ：**「日本語コメントは折り返し前提で短く」「make test（fast）は lint 非実行＝format
  ドリフトが埋もれる」**を着手前チェックに明示（後者は既に techContext に注意書きあり）。
