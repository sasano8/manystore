## 上げたいこと

「反省（retrospective／root-cause 分析）」を支える **情報整理スキル** を新設したい。
**まず情報整理に限定**し、原因分析・判断そのものは作らない＝**既存の思考フレームワーク
（品質＝[[unit-quality]] の deep think 等）へ委譲**する。横断（全 worker 共通の工程）の話。

### 関心の分離（このスキルの altitude）

flow が「deep think せよ／品質に訊け」と呼ぶだけで算法を持たないのと同じ構図にする:

- **反省スキル（new）= 情報整理レイヤ**：原因分析に必要な生事実を**決まった型で集める・並べる**だけ。
  ここで因果は結論しない。
- **分析レイヤ = 既存フレームワーク**：集まった情報に対し、品質/deep think（反証ステップ・俯瞰観点）で
  「なぜそうなったか」を導く。**反省スキルはこの算法を持たず参照するだけ**（規約の二重持ち＝drift を避ける）。
- **工程組込 = flow**：いつ発火し（コミット時/最終ゲート）、どこへ置き（`metrics/` 等）、ゲートへどう還す。

### 規定したい「情報整理の型」（分析の手前で止める）

root-cause 分析に**最低限そろえるべき生事実**を 5 区分で定型化する（中身の解釈はしない）:

1. **対象（what）**：タスク ID・コミット・*意図したスコープ*。
2. **変更の実体（facts）**：変更ファイル数・±行数・code/docs/MB の内訳・**ブラスト半径**（波及した呼出元数）。
   ＝git から厳密に取れる客観値。
3. **工程の時系列（timeline）**：意思決定点（質問と回答）・品質ゲートの実行（pass/fail/再試行回数）・
   idle 除去後アクティブ時間。＝transcript から取れる。
4. **摩擦点（friction）**：手戻りが起きた箇所（reflow ループ・再編集）・情報が見つけにくかった箇所
   （最初の編集までの Read/Grep 数）。
5. **意図 vs 実績の乖離（gap）**：*計画スコープ*と*実際の広がり*の差。←ここに「機能は局所なのに files=11」
   のような兆候が並ぶ。**並べるだけ**で、なぜかは分析レイヤに渡す。

> 出力はこの 5 区分を埋めた構造体（1 反省 1 レコード）。これがそのまま
> [[2026-06-26-workflow-metrics-and-churn-retro]] の metrics log の入力になり、deep think ゲートの素材になる。
> metrics 提案は**この情報整理の定量サブセット**＝本件が上流（型の規定）、metrics 件が下流（蓄積と還元）。

### 親への依頼

- この altitude（情報整理だけ・分析は既存 FW 委譲）でスキル化する価値があるか俯瞰判断してほしい。
- 置き場の選択：**独立スキル**（project 非依存・flow から呼ぶ）か、**flow の着手前/最終ゲートに型だけ埋め込む**か。
- 進めるなら本 repo を MVP 検証台にしてよい（手記録から）。**まず情報整理の型を固め、分析は品質 FW に任せる**方針。
